               How Batching Works Conceptually
Instead of passing messages like a hot potato the exact millisecond they arrive, 
batching introduces an ultra-fast temporary waiting room (your backend's RAM memory) and a 
traffic cop (a JavaScript timer).

The Ingestion Phase (Uncapped Speed): When Angel One floods your backend with 50 updates per second,
your WebSocket function handleTradeUpdate does nothing except overwrite a local JavaScript 
object over and over again. This operation is incredibly cheap and takes less than a microsecond.

The Throttling Phase (Regulated Speed): Completely separate from the incoming stream, 
a background clock (setInterval) fires at a steady rhythm. When the clock strikes, 
it grabs whatever the current state of the memory object is at that exact microsecond and 
blasts it to the frontend via Socket.io. Any intermediate fluctuations that happened between 
those clock cycles are discarded. The frontend only ever sees the latest snapshot at the 
moment of the broadcast.
(this simply means like, the backend keeps on recieving the live prices regulary (like every 
milisecond) and keeps it updating in a local variable or something. 
but it changes and re -renders the value in the frontend after a specific time 
range(like 2 secs or 3 secs) using setInterval)
